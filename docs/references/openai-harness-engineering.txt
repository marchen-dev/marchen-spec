OpenAI Harness Engineering 实践参考

来源: https://openai.com/index/harness-engineering/
日期: 2026-01

## 核心理念

"Agents aren't hard; the harness is hard."

Harness Engineering 是从 prompt engineering 和 context engineering 演进而来的新范式，
专注于构建让 AI agents 可靠工作的环境、约束和反馈循环。

## 五大组件

### 1. AGENTS.md Files
机器可读的配置文件，提供：
- 编码规范
- 架构约束
- 任务协议
- 仓库结构说明

### 2. 机械化约束 (Determinism via Constraints)
通过 CI/CD 中的自定义 linters 和结构性测试强制执行：
- 架构边界
- 设计不变量
- 依赖方向规则

示例：禁止 Core 包导入 CLI 包（机械化强制，而非文档约定）

### 3. 隔离环境 (Isolated Environments)
每个 agent 任务获得可复现的沙盒：
- 临时 worktrees
- 容器化构建
- 干净状态

### 4. 闭环反馈 (Closed-Loop Feedback)

不期望 agent 完美，而是构建控制系统：
  Agent 写代码 → CI 运行测试 → 错误反馈给 Agent → 自我修正

单个 Codex 运行可持续 6+ 小时（人在睡觉时）

### 5. 可观测性 (Observability)
遥测数据（logs, metrics, traces）让 agent 能够：
- 复现 bugs
- 验证修复
- 理解系统行为

OpenAI 为每个 worktree 提供临时的本地可观测性栈（Victoria Logs/Metrics/Traces）

## 知识系统设计

### AGENTS.md 作为索引，不是百科全书

❌ 失败模式：一个巨大的 AGENTS.md
  - 上下文稀缺，大文件挤掉任务本身
  - 当所有内容都"重要"，就没有内容重要
  - 瞬间腐烂，无法分辨哪些还有效
  - 无法机械验证

✅ 成功模式：AGENTS.md ≈ 100 行，只做目录
  docs/ 目录才是知识的真正载体

### docs/ 结构（OpenAI 实践）

docs/
├── design-docs/       ← 设计文档（含验证状态）
│   ├── index.md
│   └── core-beliefs.md
├── exec-plans/        ← 执行计划（一等公民！）
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/         ← 自动生成的文档（如 DB schema）
├── product-specs/     ← 产品规格
├── references/        ← 外部依赖的 llms.txt
├── DESIGN.md
├── FRONTEND.md
├── QUALITY_SCORE.md   ← 各领域质量评分，跟踪缺口
└── SECURITY.md

### 渐进式披露 (Progressive Disclosure)
Agent 从小而稳定的入口开始，被教导"去哪里找下一步信息"，
而非一开始就被淹没。

## Agent 可见性原则

"Agent 看不到的东西等于不存在"

✅ Agent 可见：
  - 代码、Markdown、Schema
  - 执行计划、设计文档
  - 日志、指标、Trace
  - CI 配置、Lint 规则

❌ Agent 不可见（= 不存在）：
  - Google Docs
  - Slack 讨论
  - 人脑中的默契

解决方案：把一切推入仓库
  Slack 对齐了架构方向？→ 写成 design-doc
  口头约定了编码规范？→ 编码为 lint 规则

## 文档 vs 机械化执行

信念层级（从弱到强）：

  文档（写下来）
     ↓ 不够
  约定（团队口头共识）
     ↓ 不够
  Lint 规则（机械检查）
     ↓ 最可靠
  结构性测试（架构约束自动验证）

OpenAI 的核心信念：
"如果不能机械化强制，它终将在生产中被打破"

## 技术债 = 垃圾回收

早期：每周五 20% 时间清理 "AI slop" → 不可扩展

后来：
  "Golden Principles" 写入仓库
       ↓
  定期后台 Codex 任务扫描偏差
       ↓
  自动开重构 PR
       ↓
  大多数 < 1 分钟 review，自动合并

持续的垃圾回收，而非定期大扫除。

## 工程师角色转变

传统工程师                    Harness 工程师
─────────────────────────────────────────────
写代码                    →   设计环境
实现功能                  →   指定意图
修 Bug                    →   建反馈循环
Code Review               →   Agent-to-Agent Review
清理技术债                →   编码 "Golden Principles"

## Merge 哲学转变

高吞吐环境下：纠错成本低，等待成本高

- 最小化阻塞 merge gates
- PR 短生命周期
- Test flakes 用后续运行修复，而非无限期阻塞

这在低吞吐环境下是不负责任的，但在 agent 驱动环境下是正确权衡。

## "无趣技术"更适合 Agent

"Boring" 技术更易被 agent 建模，因为：
- 可组合性
- API 稳定性
- 训练集中的表示

OpenAI 甚至会自己重新实现库的子集，而非引入不透明的第三方依赖，
因为 agent 需要能完全理解和推理代码。

## 关键洞察

1. 文档驱动 vs 自动化驱动是一个光谱，不是二选一
2. 小项目可以从文档开始，随规模增长逐步机械化
3. "Agent 看不到 = 不存在"是设计决策的试金石
4. 持续小增量清理技术债，而非定期大扫除
5. 高吞吐改变了 merge 哲学：纠错便宜，等待昂贵
