OpenSpec 功能参考（v1.0）

核心工作流：
propose → design → specs → tasks → implement → verify → archive

命令清单：
- opsx:propose - 创建完整提案（一步生成所有 artifacts）
- opsx:new - 创建新变更
- opsx:continue - 继续变更（创建下一个 artifact）
- opsx:ff - 快进生成所有 artifact
- opsx:apply - 执行任务
- opsx:verify - 验证实现
- opsx:sync - 同步 delta specs 到 main specs
- opsx:archive - 归档变更
- opsx:bulk-archive - 批量归档
- opsx:onboard - 引导式入门
- opsx:explore - 探索模式

### 核心模块：artifact-graph

OpenSpec 最核心的模块，负责 artifact 的依赖管理和状态追踪。

#### Schema 定义（schemas/spec-driven/schema.yaml）

```yaml
name: spec-driven
version: 1
description: Default OpenSpec workflow - proposal → specs → design → tasks
artifacts:
  - id: proposal
    generates: proposal.md
    template: proposal.md
    instruction: |
      Create the proposal document...
    requires: []

  - id: specs
    generates: "specs/**/*.md"      # glob 模式，匹配多文件
    template: spec.md
    instruction: |
      Create specification files...
    requires: [proposal]

  - id: design
    generates: design.md
    template: design.md
    instruction: |
      Create the design document...
    requires: [proposal]

  - id: tasks
    generates: tasks.md
    template: tasks.md
    instruction: |
      Create the task list...
    requires: [specs, design]

apply:
  requires: [tasks]               # 实现阶段的前置条件
  tracks: tasks.md                # 进度追踪文件
  instruction: |
    Read context files, work through pending tasks...
```

关键设计：
- 每个 artifact 声明 `requires`（依赖）和 `generates`（产出文件）
- `generates` 支持 glob 模式（如 specs/**/*.md）
- `apply` 段定义实现阶段的前置条件和进度追踪
- 每个 artifact 有独立的 `instruction`（给 LLM 的指导）和 `template`（文件模板）

#### ArtifactGraph 类（graph.ts）

```typescript
class ArtifactGraph {
  // 创建方式
  static fromYaml(filePath: string): ArtifactGraph
  static fromYamlContent(yamlContent: string): ArtifactGraph
  static fromSchema(schema: SchemaYaml): ArtifactGraph

  // 查询
  getArtifact(id: string): Artifact | undefined
  getAllArtifacts(): Artifact[]
  getName(): string

  // 核心能力
  getBuildOrder(): string[]                        // Kahn 拓扑排序
  getNextArtifacts(completed: CompletedSet): string[]  // 就绪的 artifacts
  isComplete(completed: CompletedSet): boolean     // 是否全部完成
  getBlocked(completed: CompletedSet): BlockedArtifacts  // 阻塞信息
}
```

#### 状态检测（state.ts）

```typescript
// 通过检查文件是否存在来判断 artifact 是否完成
function detectCompleted(graph: ArtifactGraph, changeDir: string): CompletedSet
// 支持 glob 模式（如 specs/**/*.md 用 fast-glob 匹配）
```

#### 指令加载（instruction-loader.ts）

```typescript
interface ArtifactInstructions {
  changeName: string
  artifactId: string
  schemaName: string
  changeDir: string
  outputPath: string          // 输出路径
  description: string
  instruction: string         // schema 中的指导文本
  context: string             // 项目上下文（给 LLM 的约束，不写入文件）
  rules: string[]             // artifact 规则（给 LLM 的约束）
  template: string            // 文件模板内容
  dependencies: DependencyInfo[]  // 依赖 artifacts 的路径和状态
  unlocks: string[]           // 完成后解锁的 artifacts
}

interface ChangeStatus {
  changeName: string
  schemaName: string
  isComplete: boolean
  applyRequires: string[]     // 实现阶段需要的 artifacts
  artifacts: ArtifactStatus[] // 每个 artifact 的状态
}

type ArtifactStatus = {
  id: string
  outputPath: string
  status: 'done' | 'ready' | 'blocked'
  missingDeps?: string[]      // blocked 时的缺失依赖
}
```

### CLI 与 Skill 的分工

```
CLI（openspec 二进制）:
  ├── openspec new change <name>          # 创建变更目录
  ├── openspec status --change <name> --json
  │     → 返回 ChangeStatus（每个 artifact 的 done/ready/blocked）
  ├── openspec instructions <artifact-id> --change <name> --json
  │     → 返回 ArtifactInstructions（模板+指导+上下文+依赖）
  ├── openspec change list                # 列出变更
  ├── openspec change show <name>         # 展示变更详情
  ├── openspec change archive <name>      # 归档
  └── openspec change verify <name>       # 校验

Skill（Claude Code prompt）:
  ├── opsx:continue 的工作流:
  │     1. openspec status --json → 拿到状态
  │     2. 找到第一个 status: "ready" 的 artifact
  │     3. openspec instructions <id> --json → 拿到指令
  │     4. 读依赖 artifacts 作为上下文
  │     5. LLM 生成内容 → 写入 outputPath
  │     6. 展示进度，提示用户继续
  │
  ├── opsx:propose 的工作流:
  │     1. openspec new change <name>
  │     2. 循环: status → instructions → 生成 → 写入
  │     3. 直到所有 applyRequires 的 artifacts 完成
  │
  └── opsx:apply 的工作流:
        1. 读 tasks.md，找未完成的 task
        2. 逐个实现，标记 [x]
```

核心设计原则：CLI 是大脑（决策），Skill 是手（执行）。
- CLI 负责：schema 解析、依赖图计算、状态检测、指令组装
- Skill 负责：调 CLI 获取信息、用 LLM 生成内容、写入文件

### MarchenSpec 需要补齐的能力

要让 skill 能驱动 MarchenSpec，核心缺两个 CLI 命令：

1. `marchen status <name> --json`
   - 返回每个 artifact 的状态（done/ready/blocked）
   - 需要 artifact-graph 模块支持

2. `marchen instructions <artifact-id> --change <name> --json`
   - 返回模板 + 指导 + 依赖信息
   - 需要 schema 中定义 instruction 字段
