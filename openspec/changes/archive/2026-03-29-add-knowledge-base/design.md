## Context

项目当前使用 OpenSpec 管理变更流（proposal → design → tasks → specs → archive），CLAUDE.md 存放开发约定，`~/.claude/memory/` 存放 Claude 会话级碎片记忆。缺少一个持久化的"全局认知"层来存放项目进展、调研结论和战略方向，导致每次新对话都需要重新调研。

## Goals / Non-Goals

**Goals:**
- 建立一个结构化的知识库目录，让 AI 能快速恢复项目上下文
- 通过模板机制确保知识库内容格式一致
- 通过 CLAUDE.md 引用建立 AI 的查阅路径
- 知识库与 openspec 变更流互补但独立

**Non-Goals:**
- 不自动化知识库更新（保持手动触发 + AI 提醒模式）
- 不为知识库构建 CLI 命令或程序化接口
- 不重复 CLAUDE.md 已覆盖的开发约定内容
- 不记录可从代码或 git 历史推导的信息

## Decisions

### 1. 知识库作为独立顶层目录

**决定**: 在项目根目录创建 `knowledge/`，与 `openspec/` 平级。

**替代方案**: 放在 `openspec/` 下作为子模块。

**理由**: 知识库的生命周期与变更流不同——知识是持续演进的，不是"完成就归档"的。独立目录体现这种本质差异，避免概念混淆。

### 2. overview.md 作为速读入口

**决定**: `overview.md` 定位为"始终保持最新的项目快照"，AI 新对话的第一读。

**替代方案**: 用 _index.md 做入口，或让 AI 扫描整个目录。

**理由**: 一个精简的全景文件（控制在 100 行以内）比让 AI 自行拼凑上下文效率高得多。覆盖式更新（而非追加）确保信息始终是当前状态。

### 3. 模板驱动格式一致性

**决定**: 在 `_templates/` 下放置 research、decision、progress-entry 三种模板，CLAUDE.md 中规定"写入前必须先读对应模板"。

**替代方案**: 在 CLAUDE.md 中用文字描述格式要求。

**理由**: 模板文件是格式的"源代码"——AI 读模板再填充比解析文字描述更可靠，也更容易维护和演进。

### 4. 索引文件机制

**决定**: research/ 和 decisions/ 各有一个 `_index.md`，每条一行摘要 + 文件链接。

**替代方案**: 不设索引，让 AI 直接 glob 目录。

**理由**: 索引文件让 AI 能快速判断是否有相关调研/决策，不需要逐个打开文件。一行摘要就够做"要不要深入"的判断。

### 5. CLAUDE.md 集成方式

**决定**: 在 CLAUDE.md 新增"知识库"章节，包含三部分：路径说明、写入规范、更新时机指引。

**理由**: CLAUDE.md 是每次对话自动加载的，是天然的入口索引。保持引用简短（不超过 20 行），避免撑大 CLAUDE.md。AI 从这里知道知识库在哪、怎么写、什么时候该更新。

## Risks / Trade-offs

**知识库过时** → overview.md 设定明确的更新时机（change 归档时、roadmap 调整时），CLAUDE.md 指引 AI 在发现过时时主动提醒。

**模板僵化** → 模板定义核心 section 但不限制额外内容，保留灵活性。

**与 Claude memory 系统重叠** → 知识库存放"值得长期保留的结构化认知"，Claude memory 存放"会话级碎片"。CLAUDE.md 中不需要特别区分，因为知识库是项目级的（跟 git 走），memory 是用户级的。
