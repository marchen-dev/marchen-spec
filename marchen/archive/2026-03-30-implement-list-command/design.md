## 背景

项目已实现 `init` 和 `new` 命令，但用户无法查看已有变更。fs 包已提供 `listDir` 和 `readYaml`，`ChangeMetadata` 类型定义在 shared 包中。本次开发遵循已有的架构模式，实现简洁。

## 目标 / 非目标

**目标：**
- 实现 `marchen list` 命令展示所有 open 变更
- 遵循已有架构模式（CLI → core → fs）
- 优雅处理边界情况（未初始化、无变更、元数据缺失）
- 提供清晰的格式化输出，使用相对时间戳

**非目标：**
- 过滤或排序选项（后续按需添加）
- 显示已归档的变更（当前只展示 open 变更）
- 详细的 artifact 状态信息（留给未来的 `status` 命令）

## 技术决策

### 决策 1：元数据文件缺失的处理方式
**选择**：跳过缺失 `.metadata.yaml` 的目录，并显示警告

**理由**：`list` 是只读命令，应当容错。单个损坏的目录不应导致整个列表失败。用户可根据警告自行排查。

**备选方案**：
- 整个命令报错退出 → 过于严格，用户体验差
- 静默跳过 → 可能掩盖潜在问题

### 决策 2：时间显示格式
**选择**：相对时间（如「2 小时前」「3 天前」）

**理由**：对于近期变更更直观。用户关心的是"最近"，而非精确时间戳。

**实现**：简单的辅助函数，无需外部依赖：
```typescript
function timeAgo(isoDate: string): string {
  const seconds = Math.floor((Date.now() - new Date(isoDate).getTime()) / 1000)
  if (seconds < 60) return '刚刚'
  if (seconds < 3600) return `${Math.floor(seconds / 60)} 分钟前`
  if (seconds < 86400) return `${Math.floor(seconds / 3600)} 小时前`
  return `${Math.floor(seconds / 86400)} 天前`
}
```

**备选方案**：
- 绝对时间戳 → 不够直观
- 外部库（dayjs）→ 不必要的依赖

### 决策 3：archive 目录处理
**选择**：扫描时按名称排除 `archive` 目录

**理由**：`archive` 是系统保留目录，不是变更。按名称过滤简单且明确。

**实现**：在 `listDir` 结果中过滤掉 `'archive'`。

### 决策 4：排序方式
**选择**：按创建时间降序（最新的在前）

**理由**：用户通常关注最近的变更。与常见 CLI 工具行为一致（git log、ls -t）。

## 风险 / 权衡

**[风险]** 元数据文件缺失可能意味着数据损坏或手动编辑
→ **应对**：显示警告并附带目录名，便于用户排查

**[权衡]** 相对时间在几天后精度下降
→ **可接受**：对于较旧的变更，「5 天前」与「6 天前」差异不大。后续可通过 `--verbose` 显示绝对时间。

**[权衡]** 初始版本无过滤/排序选项
→ **可接受**：先做简单版本，按需添加 flag。大多数项目的活跃变更不超过 20 个。
